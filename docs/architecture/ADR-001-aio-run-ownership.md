# ADR-001: Aio Run Ownership and Lifecycle

**Status:** Accepted (R1.1)
**Date:** 2026-06-28
**Phase:** R1 — Durable Run Foundation
**Implementation branch:** `feat/r1-durable-run-foundation`
**Supersedes / extends:** `docs/architecture/aio_run_event_protocol.md`,
`docs/architecture/aio_runtime_architecture.md` (both descriptive of current
state; this record locks the R1 decisions on top of them).

## Context

Aio is a consumer agent product. A "run" is one user task: a chat turn, a Deep
Research query, an image creation. Today a run is mostly a live stream: the
`HermesEventMapper` converts Hermes runtime events into `AioRunEvent` values,
`run-event-writer.ts` publishes them over an AI SDK stream, and the frontend
renders them. Nothing about a run is durable.

Verified gaps in the current code that this decision must resolve:

- `apps/web/src/lib/aio/runs/aio-run-events.ts` defines the event union but has
  **no envelope**: there is no `schemaVersion`, no `sequence`, no globally
  unique `id`, no `source`, no `receivedAt`, and no Hermes adapter metadata.
- `AioRunStatus` has no `cancelling` transient state.
- `apps/web/src/lib/aio/hermes/hermes-event-mapper.ts` drops unknown Hermes
  events in its `default` branch (`return []`) and keys tool-call identity on
  positional state (`runningToolIds`, `activeCodeExecTaskId`) instead of a
  stable Hermes identifier, so a replay is not idempotent.
- No run, event, or tool-call state is persisted. A browser refresh, a dropped
  stream, or a backend restart loses the run.

R1 makes a run durable. This ADR locks the seven decisions the rest of R1
(R1.2-R1.7) depends on. It does not choose paid infrastructure, queue
technology, a vector store, or a multi-agent model; those remain deferred.

## Decision

### 1. Aio owns the run identity

- The **Aio run ID** is a UUID generated by the Aio control plane. It is the
  primary product identity for a run and the only run identifier the UI ever
  sees.
- The Aio run row is created **before** the Hermes execution call, in the
  run orchestrator (R1.5), so the run exists and is durable even if Hermes
  never returns.
- Hermes identifiers are **adapter metadata**, not identity. `hermes_run_id` and
  `hermes_session_id` are stored on the `aio_runs` row and on event envelopes
  only as `hermes: { runId?, eventId? }`. They never appear as the run identity
  in product code or UI.

**Consequence:** swapping Hermes for another runtime later changes only the
adapter under `apps/web/src/lib/aio/hermes`. The run row and its persisted
events are stable and runtime-agnostic.

### 2. Hermes run/session mapping

- The mapping is **one Aio run to one Hermes run** for the current single-agent
  model.
- The mapping is attached after Hermes execution starts, by the orchestrator,
  via the run repository's "attach Hermes identity" operation. It is never
  written by the browser.
- Hermes does not own Aio run state. If Hermes and Aio disagree about outcome,
  Aio's persisted run status wins; Hermes output is mapped into Aio events.

### 3. Run state machine

`AioRunStatus` gains `cancelling`. The full machine:

```text
queued -> running
running -> waiting_approval
waiting_approval -> running
queued | running | waiting_approval -> cancelling
cancelling -> cancelled
queued | running | waiting_approval -> failed
running -> completed
```

- A run is created `queued`, then transitions to `running` once Hermes confirms
  the execution has started.
- `aio_runs.status` in Postgres is the **source of truth**. The `status` field
  embedded in a `run.created` / terminal event is an informational projection;
  no transition is ever implied by UI state alone.
- Terminal states (`completed`, `failed`, `cancelled`) are **immutable** except
  by administrative repair, which must write an explicit audit record.

### 4. Event ordering and idempotency

- A versioned envelope wraps every product event. V1 is defined in
  `aio-run-event-envelope.ts`:
  `id`, `schemaVersion: 1`, `runId`, `threadId`, `sequence`, `type`,
  `occurredAt`, `receivedAt`, `source: "aio" | "hermes" | "worker"`,
  `payload`, optional `hermes: { runId?, eventId? }`.
- `sequence` is **monotonic per run and assigned by the run-event repository at
  append time**, not by the mapper or the client. `(run_id, sequence)` is
  unique. Because one Hermes event can map to several product events (for
  example `tool.completed` plus `artifact.created`), sequence is per product
  event.
- Envelope `id` is globally unique.
- **Idempotency:** when an event carries a Hermes identity, reprocessing that
  same Hermes event yields the same envelope with no new `sequence`. This
  requires the mapper to stop deriving tool-call identity from positional state
  and instead key on stable Hermes fields (`tool_call_id`, event id). Aio-native
  events without a Hermes source use a producer-supplied idempotency key.
- Unknown Hermes events become an explicit **adapter-diagnostic** event; they
  are never dropped.

### 5. Reconnect and replay semantics

- The replay source of truth is the **persisted event envelope table in
  Postgres**, not Hermes. Aio owns replay and does not query Hermes for history.
- On reconnect the client requests `GET /api/runs/[runId]/events?afterSequence=N`.
  The server replays persisted envelopes in sequence order, then the client
  tails the live stream. Live and replayed events merge by envelope `id` /
  `sequence`; duplicates are discarded.
- A dropped stream, a browser refresh, or a client disconnect **never cancels
  the run**. The orchestrator continues on the backend and closes the run with a
  stable outcome regardless of transport.
- `occurredAt` (when the event happened at its source) and `receivedAt` (when
  Aio persisted it) are both stored. Replay orders by `sequence`, not by
  timestamp, so clock skew cannot reorder events.

### 6. Cancellation semantics

- The user cancels via `POST /api/runs/[runId]/stop`. The call is **idempotent**.
- `stop` is accepted from `queued | running | waiting_approval`. It transitions
  the run to `cancelling`, records `cancel_requested_at`, and calls Hermes stop.
- On Hermes confirmation the run becomes `cancelled`. If Hermes had already
  reached a terminal state first, the run keeps that state and the `stop` is a
  no-op.
- `stop` on an already-terminal run returns a stable `ALREADY_TERMINAL` code,
  not a hard error, so retries are safe.

### 7. Retention boundary

- For the private beta, runs and events are retained per user under RLS, with no
  partitioning or archival.
- **No partitioning or TTL is chosen now.** Archival is deferred to a measured
  scale trigger, recorded as a follow-up ADR: for example a documented
  per-user event row count or per-run event count crossing a threshold.
- The schema keeps `sequence` stable so that any future archival preserves
  replay order.
- Tool outputs and secrets are **redacted at persist time** (R1.2), so retention
  growth is bounded by product event count, not by raw payload size.

## Relationship to the live stream

The envelope is the **durable contract**. The existing AI SDK stream
(`data-aio-*` and the legacy `data-hermes-*` parts) remains the **live
transport**. R1 makes the orchestrator persist every envelope and publish it to
connected clients; the stream is not removed and legacy compatibility is kept.

## Non-Goals

- Multi-agent or visible agent teams: deferred.
- Workflow canvas and Agent Builder: deferred.
- BullMQ as a locked queue choice: not accepted yet.
- Qdrant: deferred; Postgres/pgvector retained.
- Replacing Hermes: not in scope; only the adapter boundary is fixed here.

## Consequences

- The mapper and event union change. Adding `cancelling` is an additive change
  to the status union; the UI already renders the other six states.
- Sequence assignment moves server-side into the repository, so the mapper
  produces envelope payloads without sequence and the repository stamps it
  transactionally.
- Replay reads from Postgres, adding a small read latency on reconnect. This is
  accepted in exchange for durability and correctness.
- Every persisted payload must be safe to store indefinitely (redacted,
  size-bounded), which constrains what the mapper may pass through.

## Acceptance

- No run-state transition is implied only by UI state.
- Terminal states are immutable except administrative repair with an audit
  record.
- A run survives refresh, a dropped stream, and a backend restart, and replays
  with no duplicate or lost events.
- The browser never inserts or mutates a lifecycle row directly.
